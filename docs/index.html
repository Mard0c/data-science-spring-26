1. Background and Problem Statement

At the outset of this project, an analysis of hospital density across the German federal states revealed a substantial decline in the number of hospitals in recent years. At the same time, healthcare planning has increasingly shifted toward concentrating inpatient bed capacity in fewer, larger hospitals, rather than maintaining a more dispersed hospital structure (Deloitte, 2024).

Against this backdrop, the project examines whether projected developments in hospital supply are consistent with this shift in bed capacity and whether the figures align with projected demand across different federal states. By comparing projected inpatient bed demand with projected bed capacity across multiple states, the study aims to identify regions that may face future shortages.

The project is guided by the following research question:
Which German cities should be prioritized for future hospital-site expansion based on projected inpatient bed demand relative to projected bed capacity?

2. Methods

2.1 Data sources and preprocessing

Hospital-level data were obtained from the German Hospital Quality Report (Krankenhausqualitätsbericht) provided by the Federal Ministry of Health. To analyse regional patterns at the level of the German federal states, hospital records were aggregated annually by Bundesland. For this purpose, a supplementary dataset linking postal codes and geographic areas to the corresponding federal states was used. In the first step, the number of hospitals was summed for each state and year. To account for differences in population size, this value was divided by the annual state population, based on data from the Statistisches Bundesamt. This produced a density measure of hospitals relative to population size.

2.2 Estimation of bed capacity and demand

In a second step, the same aggregation procedure was applied to the number of hospital beds rather than the number of hospitals. Bed capacity was therefore calculated as the annual total number of beds per federal state. To estimate demand, the analysis used the number of inpatient cases or stays. In addition, stationary and partially stationary cases were combined with the average length of stay in order to derive a measure of required bed occupancy over time.

2.3 Forecasting approach

Based on these historical indicators, forecasting models were used to project future developments in both bed supply and bed demand. The aim was to estimate regional trends for the period 2025 to 2040. Because the available time series are relatively short, the forecasting approach was intentionally kept simple and interpretable. This is important because limited numbers of observations reduce the reliability of more complex models and constrain predictive performance.

2.4 Population-adjusted gap analysis

To make projected supply-demand gaps comparable across federal states of different sizes, an additional normalization step was performed. First, state-level population data were also projected for the years 2025 to 2040.

The projected gap between bed demand and bed capacity was then divided by the projected population of the respective state and multiplied by 10,000. This yielded a standardized indicator of shortage or surplus per 10,000 inhabitants, allowing direct comparison between states.

2.5 Scope and limitations of the forecast horizon

All forecast values cover the period from 2025 to 2040. However, these projections should be interpreted primarily as a short- to medium-term analytical tool rather than as precise long-term predictions. Given the limited historical data available, forecast uncertainty increases substantially over time. More extensive time series would enable the use of more robust datasets and potentially more advanced forecasting methods. Therefore, the projections in this study are intended mainly to illustrate possible future developments and to support a conceptual assessment of regional risk patterns.

5. Reflection on the Programming Process

This section provides a technical overview of the project, including its architecture, database schema, and key programming decisions.

Run

uv run python -m src {year} {optional: path} {optional: pattern}

Test

uv run -m pytest

Architecture

Sources:

  • hospital data (data for 2018 and 2020 were corrupted on the website)
  • postcode to bundesland (csv from ?)
  • population per bundesland since 2015 (csv wide table from ?)

Ingestion:

  • hospitalxml.py -> parses xml to python dict
  • databasecontroller.py -> dict to pyarrow, then written to duckdb database.db
  • population.py -> melt csv with pandas then write frame to database.db
  • __main__.py -> parses arguments and orchestrates ingestion

If the postcode moved again, the xml file where postcode was not found gets written to ./failed for an autopsy, otherwise polite error messages are the norm.

Duckdb Tables

dataset

Primary dataset.

  • Datum: Date
  • IK: Institution ID (varchar)
  • Ort: City/Location (varchar)
  • Postleitzahl: Post code (bigint)
  • Anzahl_Betten: Number of beds (bigint)
  • Fallzahlen: A structured column containing:
    • Ambulante_Fallzahl: Outpatient cases (bigint)
    • Teilstationaere_Fallzahl: Day patient cases (bigint)
    • Vollstationaere_Fallzahl: Inpatient cases (bigint)
  • Bundesland: State (varchar)

plz_ort_bundesland

A lookup table mapping locations to states.

  • Ort: City/Location (varchar)
  • Plz: Zip code (integer)
  • Bundesland: State (varchar)

populations

Demographic data per state and year.

  • Bundesland: State (varchar)
  • year: Year (date)
  • population: Total population (bigint)

Key Decisions

  • Duckdb -> beginner friendly, portable, works nicely with pandas.
  • Xmltodict -> I forgot xml.etree existed until it was too late, xml.etree would have made life easier, as it is designed for navigating large xml structures, while xmltodict forced me to define paths in a rather cumbersome way without the option of "find x in tree".
  • Pyarrow -> keep parsing in memory (xml -> dict -> pyarrow -> duckdb is very clean and performant).
  • Pandas for ingestion and analysis -> familiar, easier to learn (more resources) and well integrated with duckdb (I did look into polars though).
  • Csv for analysis -> it was easier to work with than writing to, and getting dataframes directly from the database, easier to read than json, parquet would have been overkill, and since they were generated by pandas, the shortcomings of cvs were less relevant.
  • Plotly -> convenient and pretty, I especially liked that their website demoed most figures (with code snippets!), I wish it wasn't so commercialised.
  • Html -> easy integration with plotly, and relatively straightforward pdf export.
